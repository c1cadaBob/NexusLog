```
延迟分解:

                                                               500ms│

                                                              ┌─────────┐
                                                              │  查询   │
                                                              └─────────┘

总延迟预算: 850ms (P95)
```

**瓶颈识别**:

| 环节 | 预期延迟 | 瓶颈风险 | 优化策略 | 相关模块 |
|------|----------|----------|----------|---------|

## 3. 技术选型

### 3.1 选型原则

1. **成熟稳定**: 优先选择经过生产验证的技术
2. **社区活跃**: 有活跃的社区支持和持续更新
3. **团队熟悉**: 团队有相关技术经验
4. **可扩展性**: 支持水平扩展，满足未来增长
5. **成本可控**: 许可证、运维、学习成本可接受

### 3.2 核心技术选型对比

**消息队列选型**:

| 维度 | Kafka | RabbitMQ | Pulsar | 选择 |
|------|-------|----------|--------|------|
| 吞吐量 | 100万+/秒 | 10万/秒 | 100万+/秒 | Kafka ✅ |
| 延迟 | 毫秒级 | 微秒级 | 毫秒级 | - |
| 持久化 | 强 | 中 | 强 | Kafka ✅ |
| 运维复杂度 | 中 | 低 | 高 | RabbitMQ ✅ |
| 社区生态 | 丰富 | 丰富 | 一般 | Kafka ✅ |
| 团队经验 | 熟悉 | 熟悉 | 不熟悉 | Kafka ✅ |

**结论**: 选择 Kafka，理由：高吞吐、强持久化、团队熟悉

**存储引擎选型**:

| 维度 | Elasticsearch | ClickHouse | Loki | 选择 |
|------|---------------|------------|------|------|
| 全文检索 | 强 | 弱 | 中 | ES ✅ |
| 聚合分析 | 中 | 强 | 弱 | ClickHouse ✅ |
| 存储成本 | 高 | 低 | 低 | Loki ✅ |
| 查询延迟 | 低 | 低 | 中 | ES ✅ |
| 运维复杂度 | 中 | 中 | 低 | Loki ✅ |

**结论**: 选择 Elasticsearch，理由：全文检索能力强，查询延迟低

### 3.3 版本选择

| 组件 | 选定版本 | 最低版本 | 选择理由 |
|------|----------|----------|----------|

## 4. 关键流程设计

### 4.1 日志处理主流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        日志处理主流程                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐      │
│  │ 1.采集  │───▶│ 2.解析  │───▶│ 3.富化  │───▶│ 4.路由  │      │
│  └─────────┘    └─────────┘    └─────────┘    └─────────┘      │
│       │              │              │              │             │
│       ▼              ▼              ▼              ▼             │
│  读取日志源     格式解析       添加元数据     分发到目标         │
│  - 文件监听     - JSON解析     - 主机信息     - ES存储          │
│  - Syslog      - Grok匹配     - 地理位置     - 告警引擎         │
│  - HTTP接收    - 正则提取     - 业务标签     - 归档存储         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**流程步骤说明**:

| 步骤 | 输入 | 处理 | 输出 | 耗时 | 相关模块 |
|------|------|------|------|------|---------|
| 采集 | 原始日志 | 读取、缓冲 | 日志事件 | < 50ms | 模块1 |
| 解析 | 日志事件 | 格式解析、字段提取 | 结构化日志 | < 20ms | 模块1 |
| 富化 | 结构化日志 | 添加元数据、标签 | 富化日志 | < 30ms | 模块1,3 |
| 路由 | 富化日志 | 规则匹配、分发 | 目标队列 | < 10ms | 模块1,2 |

### 4.2 异常处理流程

**异常分类与处理**:

| 异常类型 | 触发条件 | 处理策略 | 恢复机制 | 相关模块 |
|----------|----------|----------|----------|---------|
| 网络异常 | 连接超时 > 30s | 本地缓存 + 重试 | 指数退避重连 | 模块1,9 |
| 解析失败 | 格式不匹配 | 记录原始日志 + 告警 | 人工处理 | 模块1,4 |
| 存储失败 | 写入超时/拒绝 | 重试队列 + 降级 | 自动重试 | 模块2,9 |
| 资源耗尽 | 内存/磁盘 > 90% | 限流 + 丢弃低优先级 | 扩容/清理 | 模块10 |

## 5. 接口设计

### 5.1 API 规范

- **版本控制**: URL路径版本控制 `/api/v1/`
- **认证方式**: JWT (JSON Web Token)
- **数据格式**: JSON
- **字符编码**: UTF-8
- **详细设计**: 参见 [API设计文档](./api-design.md)

### 5.2 通用响应结构

**成功响应**:
```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "timestamp": "2026-01-31T10:00:00Z",
  "request_id": "req-123456"
}
```

**错误响应**:
```json
{
  "code": 400,
  "message": "Invalid parameter",
  "error": "field 'name' is required",
  "timestamp": "2026-01-31T10:00:00Z",
  "request_id": "req-123456"
}
```

### 5.3 错误码定义

| code | HTTP状态码 | 含义 | 解决方案 |
|------|-----------|------|----------|
| 0 | 200 | 成功 | - |
| 400 | 400 | 参数错误 | 检查请求参数 |
| 401 | 401 | 未认证 | 重新登录 |
| 403 | 403 | 无权限 | 联系管理员 |
| 404 | 404 | 资源不存在 | 检查资源ID |
| 429 | 429 | 请求过频 | 稍后重试 |
| 500 | 500 | 服务器错误 | 联系技术支持 |

## 10. 配置热更新架构

所有模块统一采用配置热更新架构，无需重启服务即可生效配置变更：

```
┌─────────────────────────────────────────────────────────────┐
│                   统一配置热更新架构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  配置管理界面 → PostgreSQL → Redis → Pub/Sub → 各模块服务  │
│  (Web Console)  (版本存储)  (分发)  (通知)   (原子更新)    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**配置热更新流程**:
1. 用户在 Web Console 修改配置
2. 配置保存到 PostgreSQL，生成新版本号
3. 配置同步到 Redis，发布 Pub/Sub 通知
4. 各服务监听 Redis Pub/Sub，接收变更通知
5. 服务使用 atomic.Value 原子更新内存配置
6. 服务验证新配置，记录审计日志
