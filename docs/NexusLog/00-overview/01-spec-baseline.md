# NexusLog 当前规格基线索引（M1~M3）

> 基线日期：2026-02-28  
> 适用范围：当前项目从“前端高保真 + 后端骨架”推进到“可交付最小闭环”的执行阶段。

## 1. 这份索引的目的

本文件用于统一 `docs/NexusLog/process/05~13` 的使用方式，避免“计划、差异、需求、设计、任务、接口契约、全量蓝图”分散阅读导致执行偏差。

## 3. 推荐阅读顺序（执行导向）

1. `05`：先确认优先级和里程碑目标。  
2. `07`：确认现有差异与风险，锁定 P0 问题。  
3. `08`：确认需求边界（做什么、不做什么）。  
4. `11`：统一项目整体规划口径（范围、组织、验收门禁）。  
5. `09`：确认设计边界（谁负责什么、接口和表如何落地）。  
6. `12`：冻结接口契约（路径、字段、错误码、幂等、版本策略）。  
7. `13`：查看全项目 API 全景与后续阶段接口演进边界。  
8. `10`：按任务清单推进实现。  
9. `06`：按周执行、验收与回滚演练。

## 4. 文档间约束关系

1. `08`（需求）是 `09`（设计）、`12`（当前接口契约）、`13`（全量蓝图）和 `10`（任务）的上游。  
2. `07`（差异）中的 P0 未关闭，不得将对应里程碑标记为“完成”。  
3. `11`（整体规划）负责统一 `05/06/07/08/09/10/12/13` 的执行口径。  
4. `06`（周计划）必须映射到 `10`（任务项）和 `07`（差异状态）。  
5. 接口路径或表结构变更后，需同步更新 `05/06/07/08/09/10/11/12/13`。

## 5. 每周维护动作（建议固定在周五）

1. 更新 `10` 的任务状态：`[ ]/[~]/[x]/[-]`。  
2. 更新 `07` 的差异状态：`open/in-progress/closed` + 证据链接。  
3. 若发生范围调整，先改 `08`，再同步 `09` 和 `10`。  
4. 若发生优先级调整，先改 `05`，再同步 `06/10/11`。  
5. 若发生接口契约变更，先改 `12`，再同步 `09/10/11`。  
6. 若涉及全项目阶段范围调整，同步更新 `13` 并标注影响模块。  

## 6. 里程碑验收入口

- M1 验收：`06` 的 Week1~2 + `10` 的 M1 任务 + `07` 的相关 P0 关闭证据。  
- M2 验收：`06` 的 Week3~4 + `10` 的 M2 任务 + 接入链路追踪证据。  
- M3 验收：`06` 的 Week5~6 + `10` 的 M3 任务 + 检索/治理闭环证据。
