# NexusLog 完整开发流程（按先后顺序）

本文档给出可执行的端到端流程，目标是避免“全量瀑布式开发”，采用分阶段、可回滚、可验证的迭代方式推进。

## 阶段 0：范围收敛（先做）

1. 锁定本轮范围：只做 MVP 闭环，不并行扩展 P2 能力。
2. 明确验收标准：每个功能必须有“接口可用 + 数据可查 + 告警可见 + 可回滚”。
3. 确认路线图任务和优先级：先主链路，后增强项。

## 阶段 1：架构与契约基线

1. 冻结系统边界和服务职责（Gateway、Control Plane、API、Data Services、Agent）。
2. 先定义 API/Schema 契约，再写业务实现。
3. 统一变更分级字段和审批流程（`none` / `normal` / `cab`）。

## 阶段 2：数据模型与迁移（先于功能开发）

1. 建立基线迁移：`V1`。
2. 按能力增量迁移：`V2` 到 `V11`（状态机、分区、RLS、审计、导出、重试/死信）。
3. 所有迁移必须可前向兼容，禁止破坏性变更直接上线。
4. 先在测试环境演练迁移，再推进生产窗口执行。

## 阶段 3：核心功能垂直切片开发

1. 每次只做一条完整链路，例如“告警规则 -> 事件 -> 通知”。
2. 顺序固定：接口定义 -> 后端实现 -> 前端联调 -> 集成验证。
3. 每个切片必须带最小回滚方案（配置回滚或版本回滚）。

## 阶段 4：可观测性先行接入

1. 服务上线前先补齐 `/metrics`、`/live`、`/ready`。
2. 接入 Prometheus 抓取、Grafana 面板、Alertmanager 路由。
3. 告警先上最小闭环规则：可用性、失败率、延迟、积压。
4. 监控配置优先采用热更新，避免非必要重启。

## 阶段 5：发布与变更控制

1. 发布流水线顺序：静态校验 -> 部署 -> 热加载 -> 发布后验证。
2. 热更新失败自动回滚到上一稳定版本。
3. 生产变更默认人工审批，保留操作审计和版本记录。
4. 权限最小化：CI 用户只拥有必要的 sudo 与目录写权限。

## 阶段 6：测试与质量门禁

1. 单元测试：覆盖关键业务规则和状态机转换。
2. 集成测试：验证服务与数据库、消息中间件交互。
3. E2E 测试：覆盖核心用户路径与关键异常路径。
4. 属性测试：验证配置完整性、Schema 兼容性、CI 覆盖性。
5. 安全扫描：镜像扫描 + 代码扫描进入 CI 阻断规则。

## 阶段 7：上线与运维闭环

1. 发布后 30 分钟重点观察：错误率、延迟、队列积压、告警噪音。
2. 出现异常按 Runbook 执行：先热回滚，再人工兜底重启。
3. 事故结束后做复盘：根因、修复项、预防项、责任到人和截止时间。

## 阶段 8：进入下一轮迭代

1. 将本轮遗留问题回写到路线图任务。
2. 依据业务价值和风险，确定下一轮是 P1 收敛还是 P2 增强。
3. 重复同一流程，不跳过“可观测性、测试、回滚、审计”四个门禁。

## 一句话流程图

1. 范围收敛  
2. 架构与契约冻结  
3. 数据迁移先行  
4. 垂直切片开发  
5. 可观测性接入  
6. 发布与审批  
7. 测试门禁  
8. 上线运维与复盘  
9. 进入下一迭代
