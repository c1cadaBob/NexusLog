## 4. M1~M3 总体目标

### 4.1 M1（Week1~Week2）：认证与入口一致性

目标：把“能登录、能鉴权、能回滚”打通。

必须达成：

1. 认证闭环接口可用（注册、登录、刷新、退出、重置密码）。
2. 网关路由、白名单、upstream 与实际服务一致。
3. 迁移入口统一并完成 `up/down` 演练。

### 4.2 M2（Week3~Week4）：接入与可追溯闭环

目标：把“拉取 + 增量包 + 回执 + 死信重放”打通。

必须达成：

1. `control-plane` 提供拉取源、任务、包、回执、重放接口。
2. 远端 Agent 主路径支持增量读取并开放 `pull/ack` API，由日志服务器按任务定时主动拉取。
3. ACK/NACK、checkpoint、dead-letter 完整落库并可追溯。
4. 远程 Agent 生命周期最小闭环可用（注册、心跳、离线判定），并完成 API Key 鉴权与双密钥轮换验证。
5. 远端安装满足最小环境，不依赖编译工具，支持“预编译二进制 + systemd”与“Docker”双模式。

### 4.3 M3（Week5~Week6）：检索与治理闭环

目标：把“用户真正可用的业务面”打通。

必须达成：

1. 检索页去 mock，接入 `query-api` 实际查询。
2. 审计、告警、用户、角色形成最小治理闭环。
3. Dashboard 通过 BFF 聚合接入真实业务字段。
4. 分析链路最小闭环可用（`trend/anomaly/pattern` + 前端分析页面去 mock）。

## 5. 业务范围与非范围

### 5.1 本轮范围（P0）

1. 认证与会话闭环。
2. 网关统一路由与鉴权一致性。
3. 接入控制面与 Agent 增量闭环。
4. 检索实时/历史/收藏闭环。
5. 审计/告警/用户角色最小治理闭环。
6. 配置中心动态发布与失败回滚。

### 5.2 非范围（本轮不做）

1. P2 能力（ML/NLP、边缘计算、Service Mesh 深化）。
2. 大规模新增页面或大改 UI 信息架构。
3. 全量 IAM 深度联动替换（Keycloak 先保持后续阶段集成）。

## 8. API 与数据总体规划（高层）

### 8.1 API 分层

1. 统一入口：Gateway（`/api/v1/*`）
2. 认证与治理：`api-service`
3. 接入控制：`control-plane`
4. 查询能力：`query-api`、`audit-api`
5. 聚合入口：`bff-service`

### 8.2 数据分层

1. PostgreSQL：元数据、状态机、审计与配置版本。
2. Elasticsearch：日志正文检索。
3. Redis：热点缓存、配置通知、短周期状态。

### 8.3 迁移策略

1. 单一执行入口：`storage/postgresql/migrations`
2. 优先复用：`000012~000015`
3. 可选扩展：`000016_alert_rule_versions`（容量不足可顺延 P1）
4. 统一执行命令：`make db-migrate-*`（底层脚本：`scripts/db-migrate.sh`）
5. 执行态基线：`17-migration-execution-state-baseline.md`（区分文件态与环境执行态）

## 9. 组织与协作机制

| 角色 | 主要职责 | 默认投入 |
|---|---|---|
| FE | 页面去 mock、鉴权路由、E2E 主链路 | 2~4 人天/周 |
| BE | API 设计实现、网关联调、幂等与补偿 | 6~9 人天/周 |
| QA | 场景测试、回归门禁、验收证据 | 2~3 人天/周 |
| DBA | 迁移执行、索引评估、回滚演练 | 1~2 人天/周 |
| DevOps | 灰度发布、监控看板、回滚执行 | 1 人天/周 |

协作规则：

1. 每周冻结一次接口字段，避免跨周反复返工。
2. 每周更新 GAP 状态，未关闭 P0 不得宣告里程碑完成。
3. 所有“已完成”项必须有证据链接（PR、测试、日志、截图）。

## 10. 统一门禁与验收策略

### 10.1 周级门禁

1. 页面主路径不依赖 mock 数据。
2. 新增 API 有契约、有实现、有最小测试。
3. 涉及迁移变更必须完成 `up/down` 演练。
4. 发布变更必须具备可执行回滚动作。

### 10.2 里程碑门禁

1. M1：认证闭环 + 网关一致 + 迁移可回滚。
2. M2：拉取/增量/回执/死信重放全链路可追溯，远程 Agent 生命周期最小闭环可用，且 Agent 采用最小安装双模式并完成 API Key 校验。
3. M3：检索 + 分析 + 审计 + 告警 + 用户角色闭环完成，核心页面去 mock。

## 11. 风险、依赖与顺延策略

| 风险项 | 触发条件 | 处理策略 |
|---|---|---|
| 路由与端口不一致 | 网关 502 或误路由 | 优先修正路由配置，必要时临时回切旧路径 |
| 接入链路积压 | Agent 拉取周期过短或批量过大导致任务积压 | 调整 `pull_interval/max_records/max_bytes` 并按 source 分片调度 |
| 检索性能波动 | ES 超时上升 | 启用缓存、缩小查询窗口、限制重排序字段 |
| Agent 版本漂移 | 远程 Agent 版本分布分叉且 ACK 不一致 | 引入升级批次与版本门禁，先冻结高风险版本 |
| API Key 泄露或轮换失败 | 拉取接口被异常访问或轮换窗口配置错误 | 立即撤销旧 key、提升 next key、强制任务重建并审计追踪 |
| 热更新与生产行为漂移 | dev watch 可运行但生产镜像启动失败 | 明确 dev/prod 编排隔离并做双路径回归门禁 |
| 团队容量不足 | 周任务无法收口 | 锁定 P0，`000016` 等可选项顺延并登记原因 |

依赖条件：

1. `docker-compose` 服务健康可用于联调。
2. PG/Redis/ES 的最小稳定性满足周验收窗口。
3. 测试资源可覆盖至少核心成功/失败路径。
4. 开发环境支持容器热更新（frontend HMR + Go/Agent watcher）。
5. 远程 Agent 网络可达性、API Key 分发与轮换方案已具备最小前置能力。

1. 任何接口路径变更必须同步更新 `08/09/10/12`。
