## 二、课题内容

### 2.1 课题背景

随着企业信息系统向微服务、容器化和云原生架构演进，日志数据的来源更加分散、数量更加庞大、结构更加复杂。传统日志系统常存在以下问题：

1. 多源日志格式不统一，采集、清洗与接入成本较高。
2. 海量日志查询效率不足，故障定位与根因分析耗时较长。
3. 告警规则分散、误报较多，缺乏统一治理与事件追踪能力。
4. 审计追踪与权限控制链路不完整，难以满足合规与责任追溯要求。

因此，本课题拟设计并实现一套企业级日志管理系统，围绕日志的采集、传输、存储、检索、分析、告警、审计与可视化展示构建完整技术闭环，形成可演示、可部署、可验证的软件系统原型。

### 2.2 主要研究与实现内容

本课题的核心内容包括以下几个方面：

1. **认证与入口层设计**
   - 设计系统用户认证、会话续期、退出登录、密码重置等基础认证流程。
   - 实现统一网关入口，对系统 API 进行统一转发、鉴权校验与错误处理。

2. **日志接入与采集控制设计**
   - 设计日志采集源管理、任务调度、主动拉取、回执确认、失败补偿与死信重放机制。
   - 实现 Agent 对文件日志的增量采集、断点续传与主动拉取接口。

3. **日志检索与分析设计**
   - 设计日志实时查询、查询历史、收藏查询、聚合统计、异常检测与聚类分析能力。
   - 实现日志查询服务与前端检索页面联动。

4. **告警与审计治理设计**
   - 设计告警规则、告警事件、静默策略、通知渠道、事故升级等治理功能。
   - 实现审计日志查询、导出、用户与角色治理、登录策略配置等安全治理能力。

5. **系统可视化与工程化交付设计**
   - 实现前端管理控制台，包括登录、检索、治理、审计、告警和首页概览等页面。
   - 完成部署、回滚、监控、发布门禁、测试脚本与文档整理，保证课题成果具备完整交付形态。

### 2.3 课题目标

1. 完成一个可运行的日志管理系统原型，形成从日志接入到前端展示的核心主链路。
2. 实现认证、采集、查询、治理、审计与告警等关键业务模块的最小闭环。
3. 形成结构完整、文档齐全、可部署、可验证、可演示的毕业设计成果。
4. 通过本课题训练学生在系统分析、架构设计、数据库设计、前后端开发、测试验证和文档编写等方面的综合能力。

## 三、明确的技术要求

### 3.1 总体技术路线要求

1. 系统采用前后端分离与模块化设计思想。
2. 前端控制台采用现代 Web 技术栈实现，后端采用多服务协同架构实现。
3. 数据存储采用“关系型数据库 + 检索型数据库”组合方式，兼顾事务一致性与日志查询效率。
4. 系统需具备本地开发部署能力，并支持通过容器化方式完成整体联调。
5. 系统应具备完整工程结构，代码、文档、测试和配置应能统一管理。

### 3.2 功能技术要求

毕业设计系统至少应满足以下功能要求：

1. **认证功能**
   - 支持注册、登录、刷新令牌、退出登录、重置密码等功能。
   - 支持受保护页面访问控制与会话状态校验。

2. **日志接入功能**
   - 支持采集源管理、任务触发、任务状态追踪、包查询、回执写入、死信重放等功能。
   - 支持 Agent 增量采集与主动拉取接口。

3. **日志查询功能**
   - 支持实时日志查询、查询历史管理、收藏查询管理。
   - 支持按关键字、时间范围、分页、排序、过滤条件进行检索。

4. **分析与概览功能**
   - 支持基础聚合统计、趋势视图、异常检测或聚类分析中的至少两类能力。
   - 支持首页概览聚合展示。

5. **治理与审计功能**
   - 支持审计日志查询与导出。
   - 支持用户管理、角色管理、登录策略、告警规则、告警事件或静默治理中的核心功能。

### 3.3 性能与可靠性要求

在教学与实验环境下，系统应尽量满足以下技术指标：

1. 登录、注册等基础认证接口应能稳定返回结果，普通请求响应时间建议控制在 `2s` 以内。
2. 常规日志检索请求在典型实验数据量下应能在 `3s` 内返回首屏结果。
3. 新日志写入后，应能在较短时间内被查询链路检索到，教学环境建议控制在 `10s` 以内。
4. 系统应支持基础异常处理，对参数错误、鉴权失败、依赖异常等场景给出明确错误提示。
5. 系统应具备基础回滚、恢复、补偿或重试能力，确保关键链路可验证、可追踪。

### 3.4 安全与规范要求

1. 用户密码不得明文存储，须采用安全散列方式保存。
2. 接口访问应包含基础身份校验与权限控制。
3. 系统需保留关键治理操作的审计记录。
4. 源代码需具备清晰目录结构和命名规范，避免无序堆叠。
5. 涉及数据库结构调整时，需通过迁移脚本统一管理。

### 3.5 工程化要求

1. 毕设作品应以完整工程项目形式提交，不得仅提交零散代码片段。
2. 需提供统一启动说明、依赖说明、部署步骤和关键配置说明。
3. 需提供必要测试脚本或验证记录，至少覆盖认证主路径、日志接入主路径和查询主路径。
4. 文档、代码、脚本、配置应保持一致，避免“说明与实现严重不符”的情况。

## 四、提交成果及具体要求

### 4.1 总体提交成果

学生完成毕业设计后，原则上应提交以下成果：

1. 毕业设计任务书 `1` 份。
2. 开题报告 `1` 份。
3. 毕业设计说明书 `1` 份。
4. 可运行的软件系统原型 `1` 套。
5. 完整源代码与配置文件 `1` 套。
6. 数据库迁移脚本与初始化脚本 `1` 套。
7. 测试报告或测试记录 `1` 份。
8. 部署说明、运行说明、回滚说明文档各 `1` 份。
9. 答辩 PPT `1` 份。
10. 演示脚本或演示说明 `1` 份。
11. 如学院要求，可补充演示视频 `1` 份。

### 4.2 毕业设计说明书具体要求

1. 说明书正文建议不少于 `18000` 字。
2. 说明书应包括以下主要内容：
   - 课题背景与意义
   - 需求分析
   - 总体架构设计
   - 数据库设计
   - 关键模块设计与实现
   - 测试与验证
   - 总结与展望
3. 说明书应做到结构完整、语言规范、图文对应、数据真实。
4. 参考文献建议不少于 `20` 篇，其中近 `5` 年文献建议不少于 `8` 篇。

### 4.3 图量与图表要求

1. 全文应包含系统设计与实现相关图示，图量建议不少于 `15` 幅。
2. 其中应至少包含以下类型图示：
   - 系统总体架构图不少于 `1` 幅
   - 业务流程图或功能流程图不少于 `3` 幅
   - 时序图 / UML 图不少于 `3` 幅
   - 数据库 E-R 图或数据结构关系图不少于 `1` 幅
   - 部署结构图不少于 `1` 幅
   - 页面截图、结果图、运行效果图不少于 `6` 幅
3. 数据表、对比表、测试结果表等表格总量建议不少于 `8` 个。

### 4.4 软件作品具体要求

毕业设计作品需满足以下数量与完整性要求：

1. 提交 **可运行系统原型 `1` 套**。
2. 至少包含以下作品组成：
   - 前端控制台 `1` 个
   - API 网关 `1` 个
   - 后端服务不少于 `6` 个
   - 日志采集 Agent `1` 个
   - 关系数据库实例 `1` 套
   - 日志检索存储实例 `1` 套
3. 系统至少覆盖以下 `8` 类核心模块中的大部分：
   - 认证模块
   - 网关模块
   - 接入控制模块
   - Agent 采集模块
   - 查询模块
   - 审计模块
   - 告警模块
   - 用户与角色治理模块
   - 首页聚合与展示模块
4. 系统至少提供 `30` 个以上可用 API 接口或等价功能接口。
5. 数据库中应包含不少于 `12` 张核心业务表或等价核心数据实体。
6. 应支持至少 `3` 条完整演示链路：
   - `登录 -> 首页/控制台`
   - `配置采集源 -> 触发接入 -> 查询日志`
   - `治理操作 -> 审计留痕 / 告警规则管理`

### 4.5 代码与工程文件要求

1. 源代码需完整提交，目录结构清晰，能够独立打开和阅读。
2. 应包含配置文件、启动脚本、数据库迁移脚本、测试脚本、依赖说明和必要资源文件。
3. 如使用第三方组件，应在文档中说明其用途与版本。
4. 不得仅提交编译产物，需同时提交源代码与文档。

### 4.6 软硬件数量与配置要求

#### 4.6.1 硬件要求

1. 开发终端 `1` 台：用于编码、调试、文档编写与本地联调。
2. 测试或部署主机 `1` 台：可为实验室服务器、台式机或高性能笔记本。
3. 云服务器 `1` 台（可选）：用于公网联调、远程演示或备份验证。
4. 日志数据源环境 `1~2` 套：可通过本地日志文件、容器日志或模拟服务产生测试日志。

建议硬件配置如下：

- CPU：`8` 核及以上
- 内存：`16GB` 及以上
- 可用磁盘空间：不少于 `100GB`

#### 4.6.2 软件环境要求

至少准备以下软件环境各 `1` 套：

1. 操作系统环境：Windows / Linux / macOS 任一开发环境 `1` 套。
2. Go 开发环境 `1` 套。
3. Node.js + pnpm 前端开发环境 `1` 套。
4. Docker + Docker Compose 容器环境 `1` 套。
5. PostgreSQL 数据库实例 `1` 套。
6. Elasticsearch 检索存储实例 `1` 套。
7. Redis 缓存实例 `1` 套。
8. Kafka 消息环境 `1` 套（如采用兼容链路，可用单节点或教学环境替代）。
9. 现代浏览器环境 `1` 套，用于前端页面调试与演示。

### 4.7 文档与答辩材料要求

1. 答辩 PPT 建议控制在 `15~25` 页。
2. 演示脚本应覆盖课题背景、系统架构、核心功能、关键链路演示与总结展望。
3. 若提交演示视频，时长建议控制在 `5~10` 分钟。
4. 应准备系统运行截图、测试结果截图和关键流程截图，用于支撑答辩陈述。
